# Микрофронтенды на Vite: `@originjs/vite-plugin-federation`

Документ фиксирует типичные **сложности** подхода Module Federation в связке с Vite и **практические выводы** для этого репозитория (host + несколько remotes: React, Vue и др.).

## 1. Экосистема и зрелость

**Сложность.** Плагин `@originjs/vite-plugin-federation` — решение сообщества, а не «официальная» часть Vite. Поведение может отличаться от эталона Webpack Module Federation; при обновлениях Vite и Rollup возможны регрессии. Для нетривиальных сценариев (SSR, нестандартные рантаймы, тонкая настройка чанков) меньше готовых рецептов, чем в экосистеме Webpack MF.

**Вывод.** Закладывать время на проверку совместимости при апгрейде toolchain; для критичных продакшен-систем иметь план отката и зафиксированные версии зависимостей.

## 2. Версии и общие зависимости (`shared`)

**Сложность.** Host и remotes обязаны согласовывать версии общих пакетов (как в `host-app/vite.config.ts`: `react`, `react-dom`, `redux`, `vue` и т.д.). Несовпадение major (часто и minor) ведёт к дублированию кода в бандле или к тонким ошибкам в рантайме: два экземпляра React, «разорванный» контекст, несовместимые хуки. Настройки `singleton` / `requiredVersion` не снимают необходимости дисциплины между командами. Смешение стеков (React + Vue) снижает выгоду от dedupe и увеличивает операционную нагрузку.

**Вывод.** Ввести явную политику версий для `shared`, проверку в CI (совпадение диапазонов в `package.json` у host и remotes), документировать обязательные пакеты в корневом `README.md` и при релизе синхронизировать обновления фреймворков.

## 3. Рантайм и производительность

**Сложность.** Загрузка идёт в несколько этапов: remote entry, затем чанки. Без prefetch, кэширования по контент-хэшу и HTTP/2 (или аналога) растёт задержка до интерактивности по сравнению с одним предсобранным бандлом. Несколько remotes умножают цепочки запросов. Пока модули не пришли, нужны осмысленные состояния загрузки и обработка ошибок сети (в host это частично закрывается `React.lazy` / `Suspense`).

**Вывод.** Проектировать раздачу статики (CDN, длинный кэш для имён с хэшем), по возможности уменьшать число критических remotes на первом экране, продумывать стратегию предзагрузки для частых маршрутов.

## 4. Операции и инфраструктура

**Сложность.** Базовые URL remotes задаются при сборке host через переменные вида `VITE_REMOTE_*` (см. корневой `README.md`): смена конечных адресов без пересборки host невозможна, если отдельно не отдавать конфиг с сервера. Несовместимый деплой remote относительно ожиданий host ломает продакшен до отката или версионирования контрактов. В разработке разные порты означают разные origin и риски **CORS**; в Docker-стенде проще выдерживать **один origin** через прокси nginx у host.

**Вывод.** Для продакшена зафиксировать схему URL (прокси под одним доменом), регламент релизов remotes и host, при необходимости — runtime-конфиг URL или согласованное версионирование артефактов.

## 5. Контракт между приложениями и DX

**Сложность.** Импорты вида `remoteName/ExposedName` — строковый контракт. TypeScript можно подкрепить декларациями (например `host-app/src/types`), но на уровне рантайма отсутствие модуля или смена сигнатуры exposed API проявляется уже у пользователя. Отладка и stack trace через границы контейнеров сложнее, чем в монолите; source maps с нескольких origin требуют внимания.

**Вывод.** Документировать публичные точки `exposes`, рассмотреть контракт-тесты или минимальные интеграционные проверки host против собранных remotes перед выкладкой.

## 6. Модель изоляции

**Сложность.** Module Federation не даёт изоляции уровня iframe: общий глобальный контекст JavaScript. Конфликты имён в CSS, полифиллы, побочные эффекты при инициализации remotes остаются зоной ответственности команд.

**Вывод.** Согласовать методологию стилей (CSS modules, префиксы, design tokens), избегать загрязнения глобальной области без необходимости.

## 7. SSR и продвинутые сценарии

**Сложность.** Серверный рендер и гидратация в связке с динамической подгрузкой remote-модулей заметно усложняют архитектуру по сравнению с классическим CSR. Не все паттерны Webpack MF переносятся один в один на Vite-плагин.

**Вывод.** Если SSR обязателен, закладывать отдельное проектирование пайплайна загрузки и совместимости, а не полагаться на конфигурацию «как для чистого CSR».

## 8. Передача общих данных между host и remote

**Сложность.** Данные могут пересекать границу по-разному: **пропсы** в корневой компонент remote, **общее состояние** (Redux, React Context), **конфигурация среды**, **внешние каналы** (URL, API, глобальные объекты, события). Каждый вариант добавляет точки согласования между командами.

- **Пропсы.** Host рендерит экспорт remote как дочерний компонент (в этом репозитории — страницы вроде `host-app/src/pages/LeagueApp`, `TaskListReact` с `React.lazy` и динамическим `import('…/…')`). Сигнатура пропсов не проверяется сборкой host без общего пакета типов или актуальных деклараций в `host-app/src/types`; смена контракта в remote без синхронного обновления host даёт ошибки в рантайме. Нестабильные ссылки (новые объекты и колбэки на каждый рендер) усиливают лишние перерисовки внутри remote.

- **Через общий пакет / общий компонент.** Host и remotes подключают один workspace- или npm-пакет (в репозитории ориентир — `packages/shared-react-ui` и аналогичные библиотеки). Из пакета импортируют презентационные компоненты, типы пропсов и при необходимости **React Context** (`Provider` / `useContext`). Host может оборачивать remote в оболочку из пакета и передавать данные пропсами вниз, в том числе в корневой компонент remote, если контракт это описывает. Для **контекста** критично, чтобы host и remote использовали **один и тот же экземпляр модуля** с определением контекста и **один экземпляр React** (см. раздел 2); иначе получаются два разных контекста и «пустые» значения в remote. При необходимости пакет включают в **`shared`** в federation наравне с `react`, чтобы не дублировать библиотеку в чанках. Упрощённый вариант — пакет экспортирует только **типы и «глупые» UI-компоненты**, а данные идут явными пропсами от host в remote; контракт централизован в версионируемой зависимости, а не только в `host-app/src/types`. Для **Vue** или иного стека общий React-пакет не заменяет мост: нужен отдельный слой или согласованное обновление версий пакета при релизах host и remotes.

- **Общий React-стек (store, контекст, роутинг).** Чтобы host и React-remote видели один и тот же Redux или контекст, нужен **один экземпляр** `react` / `react-dom` и согласованный `shared` в federation (см. раздел 2). Два экземпляра React или несовпадающие версии разрывают цепочку провайдеров. Порядок обёрток провайдеров должен покрывать и host, и remote.

- **Разные фреймворки (например React host и Vue remote).** Общий React-контекст или Redux не передаются «как есть»; нужны явные мосты: пропсы в обёртку, события, отдельный HTTP/API-слой, ограниченный общий объект-шина.

- **Конфигурация и секреты.** У каждого приложения своя сборка: `import.meta.env` и аналоги фиксируются при **сборке remote**, а не host. Передать «с host» runtime-значения без договорённости нельзя: отдельный runtime-конфиг (скрипт с сервера), пропсы в корень remote или запрос после монтирования.

- **Асинхронная загрузка.** Пока remote не загружен (`lazy` / `Suspense`), граничного компонента нет: синхронная передача данных в remote до появления модуля невозможна без промежуточного состояния в host или повторного рендера после загрузки.

**Вывод.** Зафиксировать способ обмена данными в архитектурных правилах: для пропсов — версионируемый контракт и при возможности общий пакет типов; при использовании **общего пакета компонентов** — semver и совместное обновление зависимости, проверка единственности экземпляра React и модулей контекста (`shared` при необходимости); для глобального UI-состояния — выверенный `shared` и единая иерархия провайдеров; для конфигурации — runtime-конфиг или явные пропсы/API; для смешанных стеков — отдельный слой интеграции, а не неявный общий контекст.

## Итоговые выводы

| Плюс подхода | Цена |
| --- | --- |
| Независимые сборки и деплои remotes | Согласование `shared`, URL и порядка релизов |
| Меньше дублирования при едином стеке и выверенном `shared` | Сеть, кэш, несколько round-trip до UI |
| Гибкое встраивание фич с разных команд | Строковый контракт, отладка через границы, риск поломки при несовместимом remote |
| Пропсы и общий store при одном React | Контракт пропсов, конфиг сборки remote, мосты между разными фреймворками |
| Общий пакет UI, типов и контекста | Версии пакета, дедуп через `shared`, один React для Context |

**Практическое резюме.** `vite-plugin-federation` осмысленен, когда командам нужны **автономные пайплайны** и **runtime-подключение** модулей, а организация готова платить **операционной дисциплиной** (версии, деплой, CORS/origin, документация контракта, способ передачи данных между host и remotes). Для этого репозитория ключевые точки внимания уже отражены в корневом `README.md`: блоки про `shared`, переменные `VITE_REMOTE_*`, CORS в локальной разработке и переменные окружения при сборке отдельных remotes (например league-app).
